## 1. 패키지 매니저


### 1.1 패키지 매니저란

패키지 매니저는 프로젝트에서 사용되는 라이브러리(패키지)들을 설치 / 업데이트 / 삭제 등을 관리하는 도구이다.

ex) npm, yarn, pnpm

### 1.2 패키지 매니저 역할

1. **패키지 설치**
    1. 아래와 같은 명령어로 `axios` 라는 패키지를 설치할 수 있다.
    
    ```bash
    npm install axios
    yarn add axios
    pnpm add axios
    ```
    
2. **의존성 관리**
    1. 아래와 같이 `package.json` 파일에서 패키지의 의존성을 관리한다.
        1. 의존성은 아래와 같이 패키지 이름, 버전, 스크립트(단축 명령어), dependencies 설정 등을 말한다.
    
    ```json
    {
      "name": "my-awesome-project",
      "version": "1.0.0",
      "description": "이 프로젝트에 대한 간단한 설명입니다.",
      "main": "index.js",
      "scripts": {
        "start": "node index.js",
        "test": "echo \"Error: no test specified\" && exit 1",
        "dev": "nodemon app.js"
      },
      "keywords": ["node", "javascript", "example"],
      "author": "Your Name",
      "license": "MIT",
      "dependencies": {
        "express": "^4.18.2"
      },
      "devDependencies": {
        "nodemon": "^3.0.1"
      }
    }
    ```
    
3. **lock 파일 생성**
    1. 아래와 같이 lock 파일을 생성한다.
    `package.json` 파일은 ‘이 프로젝트에서는 1~2 버전 사이의 XX 라이브러리를 사용합니다’
    `package-lock.json` 파일은 ‘이 프로젝트는 1.1 버전의 XX 라이브러리를 사용하고 XX 라이브러리는 YY, ZZ 라이브러리들을 추가로 사용합니다’
    라고 생각하면 된다.
    - npm → `package-lock.json`
    - yarn → `yarn.lock`
    - pnpm → `pnpm-lock.yaml`
4. **스크립트 실행**
    1. 프로젝트의 개발 환경을 실행하거나 다른 스크립트를 실행할 때 패키지 매니저를 통해서 실행한다.
    
    ```bash
    npm run dev
    yarn dev
    pnpm dev
    ```
    

## 2. npm


### 2.1 npm이란

npm은 Node.js 생태계의 기본 패키지 매니저로 Node.js를 설치하면 npm CLI를 기본적으로 사용할 수 있다.

### 2.2 npm 장점

1. **좋은 접근성**
    1. Node.js를 설치하면 기본적으로 함께 설치 되다 보니 사람들이 보다 많이 사용하고 진입장벽이 낮다.
2. **생태계 표준 패키지 매니저**
    1. 앞서 말한 것처럼 Node.js는 사실상 대부분의 사람들이 처음 접하는 패키지 매니저이기 때문에
    대부분의 예제나 공식 문서에서는 보통 npm을 기준으로 설명하거나 예시 코드를 작성해둔다.

### 2.3 npm 단점

1. **좋지 않은 디스크 효율**
    1. pnpm 공식 문서에서는 npm이 프로젝트 별로 의존성 복사가 많이 일어난다고 한다.
    (의존성 복사가 무엇인지, 왜 pnpm이 디스크 효율이 더 좋은지는 pnpm 파트에서 나옴)

## 3. Yarn


### 3.1 Yarn이란

Yarn은 PnP, Workspaces, 캐시 전략, 플러그인 확장성 등을 주요 기능으로 내세우는 패키지 매니저이다.

### 3.2 Yarn Classic과 Yarn Berry

- Yarn Classic은 1.x 버전을 의미한다.
- Yarn Berry는 2.x 이상의 버전을 의미하다.

이렇게 굳이 나누는 이유는 Yarn은 2.0 버전 이후로 기존 Yarn과 구조적으로 많이 변화했고,
위에서 말한 것처럼 핵심 기능들을 내세우면서 구분하게 되었다.

### 3.3 Yarn 특징

1. **Plug'n'Play (PnP)**
    1. 정통적인 node_modules 설치 대신 패키지 해석 정보를 별도로 관리하여 설치 성능과 구조적 명확성을 높였다.
2. **Workspaces**
    1. 여러 패키지를 하나의 저장소에서 관리할 수 있다.
3. **캐시 전략**
    1. Yarn은 설치한 패키지를 기본적으로 캐싱하고, 다른 프로젝트들과도 공유해 설치 속도와 디스크 사용량을 개선한다.
4. **플러그인 확장성**
    1. Yarn은 플러그인 API를 제공해서 다양한 사용자 요구를 확장 방식으로 처리할 수 있도록 설계 되었다.

### 3.4 Yarn 장점

1. **많은 고급 기능**
    1. 위에서 말한 것처럼 PnP, workspace, 확장성, 캐시 등 Yarn에서 제공하는 고급 기능이 많다.
2. **모노레포와의 적합성**
    1. Yarn의 특징을 보면 알 수 있듯이 Yarn은 큰 프로젝트 관리에 유용하기에 모노레포와 잘 맞는다.
3. **선택 가능한 설치 전략**
    1. 기본은 PnP를 베이스로 하지만, node_modules를 사용하도록 변경할 수 있다.

### 3.5 Yarn 단점

1. **높은 난이도**
    1. npm만 써본 사람 입장에서는 Yarn에서 제공하는 다양한 기능들을 학습하기에 시간이 걸리고 난이도가 높은 편이다.
2. **정보 혼란**
    1. 위에서 말한 것처럼 Yarn은 1.x 버전과 2.x 버전의 차이가 크다. 따라서 사람들이 과거의 버전의 정보를 보고 혼동할 수 있다.

## 4. pnpm


### 4.1 pnpm이란

pnpm은 성능과 디스크 효율을 강조하는 패키지 매니저로 content-addressable store와 링크를 기반으로 한 node_modules 관리를 한다.

### 4.2 pnpm 특징

1. **content-addressable store**
    1. 같은 파일 내용을 중복 저장하지 않는 저장소로 버전 차이로 파일 내용이 일부만 달라지는 경우에도
    변경된 파일 내용만 추가로 저장하는 식으로 디스크를 절약한다.
2. **Hard link / Symbolic link**
    1. 프로젝트마다 별도로 패키지를 모두 설치하는 게 아닌, 저장소의 파일을 링크 형식으로 연결하여 사용한다. 그래서 설치가 빠르고 공간을 덜 쓴다.
3. **Virtual store**
    1. `node_modules/.pnpm` 안에 가장 저장소가 있고 여기에 sotre 파일들이 연결된다.
4. **Workspace 지원**
    1. pnpm도 `pnpm-workspace.yaml` 기반으로 workspace 구성이 가능하다.

### 4.3 pnpm 장점

1. **디스크 효율**
    1. 위에서 말한 것처럼 sotre를 공유해 프로젝트마다 같은 패키지를 공유해서 사용한다.
2. **설치 속도와 재사용성**
    1. 이미 설치가 되어 있는 패키지를 링크해서 사용하기에 설치 속도가 빠르고, 재설치 비용을 줄일 수 있다.
3. **대규모 프로젝트**
    1. 아까 말한 것처럼 worckspace 구성이 가능하기 때문에 모노레포나 대규모 프로젝트에도 적합하다.

### 4.4 pnpm 단점

1. **낯선 구조**
    1. store, hard link, symlink 기반으로 높은 성능을 제공하는 것 만큼 내부 구조를 이해하기 쉽지 않다.
