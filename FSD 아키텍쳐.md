## 1. FSD 기본 개념


### 1.1 FSD란?

애플리케이션을 기능 단위로 분할하여 프로젝트가 커지더라도, 프로젝트가 구조적이며 이해하기 쉽게 하여  
유지보수성을 높이는 아키텍쳐이다.  

### 1.2 FSD 구조

FSD 구조는 아래에서 자세히 다루겠지만 여기서 간단하게 다루자면, 가장 기본적인 것부터 최상위까지 여러 계층으로 나뉘어서 사용된다.  

계층은 `app`, `processes`, `pages`, `widgets`, `features`, `entities`, `shared` 로 7가지 있다.  

여기서 `processes` 는 현재 사용되지 않기 때문에 실제로 사용되는 건 6가지이다.

## 2. 레이어 **(Layers)**


### 2.1 App

프로젝트의 가장 상위에 존재하며, 전역 설정을 담당하는 레이어이다.  
라우팅, 전역 스타일, 테마 설정 등이 여기에 포함된다.  

### 2.2 Pages

애플리케이션 내에 개별 페이지를 정의하며, `/login` 처럼 특정 URL에 매핑 되는 전체 페이지를 구성한다.  
기존 프로젝트 구조에 있는 pages 폴더를 생각하면 된다.  

### 2.3 Widgets

페이지 내에서 독립적으로 작동하는 큰 기능을 단위로 하는 레이어이다.  

ex) 검색 바, 대시보드, 사이드바 메뉴  
처럼 큰 UI 컴포넌트나 기능을 수행하므로 여러 페이지에서 사용될 수 있다.  

### 2.4 **Features**

**재사용** 가능한 비지니스 로직 기능을 위한 레이어이다.  

ex) FollowUser
처럼 독립적으로 실행될 수 있는 하나의 특정 동작을 정의한다.  

### 2.5 **Entities**

프로젝트에서 다루는 비지니스의 핵심 데이터를 나타내는 레이어이다.  

ex) User  
여기서 User는 사용자의 주요 정보(이름, 이메일, 프로필 사진) 등을 관리한다.  

### 2.6 **Shared**

프로젝트 전체에서 재사용 사용할 수 있는 유틸리티, 기본 컴포넌트, 스타일 등을 포함하는 레이어이다.  

ex) 시간 포맷 변경 유틸 함수, Button, ColorStyle  

### 2.7 레이어 규칙

FSD의 중요 규칙중 하나는 단방향 의존성이다.  

ex) `Features` 레이어는 `Shared`나 `Entities` 레이어의 구성 요소를 참조할 수 있지만, `Widgets` 레이어의 구성 요소는 참조할 수 없다.  

이 규칙을 따르면 코드 결합도를 낮춰 나중에 코드 수정이 일어나더라도 연쇄적으로  
코드 수정이 확산되는 것을 막을 수 있다.

## 3. 슬라이스 (Slices)


### 3.1 슬라이스 기본 개념

auth, admin, user 처럼 도메인 별로 폴더를 나누는 것이다.  

### 3.2 슬라이스 규칙

1. `app` , `shared` 는 슬라이스를 갖지 않는다.  
2. 슬라이스는 같은 레이너 안에서 다른 슬라이스를 참조할 수 없다.  
    1. 예를 들어 같은 레이어에 admin 슬라이스와 user 슬라이스가 있다면이 두 슬라이스는 서로 참조할 수 없다.  

## 4. 세그먼트 (Segments)


### 4.1 세그먼트 기본 개념

슬라이스 안에는 기능 별로 분리된 세그먼트들이 있다.  

- `model` : 상태 관리, 비지니스 로직등을 포함하여, 사용자 정보나 팔로우 상태 같은 데이터를 관리한다.  
- `ui` : 컴포넌트들을 포함하여 화면을 구성하는 UI 요소를 제공한다.  
- `api` : 우리 평범하게 생각하는 서버와 통신하는 api 코드가 들어있다.  
- `lib` : 유틸 함수나 헬퍼 함수가 들어간다.
