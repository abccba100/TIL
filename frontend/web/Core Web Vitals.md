## 1. Core Web Vitals 기본 개념

### 1.1 Core Web Vitals란?
구글이 "이 세 가지가 사용자 경험의 핵심이다"라고 정한 웹 성능 지표다.

SEO 랭킹에 직접 영향을 주기 때문에 단순히 "빠르게 만들기"가 아니라 비즈니스랑 연결되는 개념이다.

예를 들어 "이 페이지 느려요"는 그냥 불평이지만, "LCP가 4초라 SEO 손해 보고 있고, 고치면 검색 유입 올라갈 수 있어요"는 설득이다.

## 2. 지표 3가지


### 2.1 LCP (Largest Contentful Paint)

페이지에서 가장 큰 콘텐츠(히어로 이미지, 큰 텍스트 등)가 화면에 그려지는 시간.

사용자가 "아 페이지 떴다"고 느끼는 순간이라고 보면 된다.

- 좋음: **2.5초 이내** / 나쁨: 4초 초과
- 주요 원인: 큰 이미지 최적화 안 됨, 서버 응답 느림, 렌더 블로킹 JS/CSS

- 렌더 블로킹
    
    렌더 블로킹은 JS나 CSS를 다운로드 중일 때 브라우저가 화면 그리는 것을 멈추는 걸 말한다.
    

### 2.2 CLS (Cumulative Layout Shift)

페이지 로딩 중에 요소들이 갑자기 툭툭 움직이는 정도.

버튼 누르려는데 광고 배너가 뒤늦게 뜨면서 다른 걸 클릭하게 되는 경험이 CLS가 높은 거다.

- 좋음: **0.1 이하** / 나쁨: 0.25 초과
- 주요 원인: 이미지에 width/height 없음, 폰트 교체 시 글자 크기 변함, 동적으로 삽입되는 요소

- width/height 없는 게 문제인 이유
    
    이미지에 width/height가 없으면 브라우저에서 이미지가 다운로드 되기 전까지는 크기를 0으로 잡기 때문에 크기가 0이었다가 실제 이미지 크기로 변해서 그렇다.
    

이미지에 width/height가 없으면 브라우저가 이미지 다운로드 전까지 공간을 0으로 잡았다가, 다운로드 완료되면 갑자기 크기가 생기면서 주변 요소들이 툭 밀린다. 그게 레이아웃 이동이다.

### 2.3 INP (Interaction to Next Paint)

버튼 클릭 같은 인터랙션에 반응해서 화면이 업데이트되는 시간. 2024년에 FID를 대체했다.

- 좋음: **200ms 이내** / 나쁨: 500ms 초과
- 주요 원인: 메인 스레드 블로킹, 무거운 JS 실행, 불필요한 리렌더링

### 2.4 LCP 기준이 2.5초인 이유

INP는 200ms인데 LCP는 왜 2.5초냐고 할 수 있는데, 측정하는 대상 자체가 달라서 기준도 다르다.

LCP는 페이지를 처음 열었을 때 서버에 요청 보내고 → HTML 받고 → CSS/JS 파싱하고 → 이미지 다운로드하는 전체 과정이 포함된다. 네트워크 왕복만 해도 수백ms는 기본으로 나온다.

INP는 이미 페이지가 다 뜬 상태에서 클릭했을 때 반응 속도만 재는 거라 기준이 훨씬 빡빡한 거다.

식당으로 비유하면 음식 나오는 시간(LCP) vs 직원이 주문 받는 반응 속도(INP).

## 3. 측정 도구 비교


둘 다 성능을 측정하는 도구지만 데이터 출처가 다르다.

### 3.1 DevTools Lighthouse

- 내 컴퓨터에서 네트워크/CPU를 인위적으로 throttling해서 시뮬레이션 돌리는 거다
- 매번 돌릴 때마다 점수가 조금씩 달라질 수 있다
- 실제 사용자가 경험하는 것과 다를 수 있다

### 3.2 PageSpeed Insights

- 구글 서버에서 시뮬레이션을 돌리는 거라 내 컴퓨터 환경 영향을 안 받는다
- 방문자가 충분하면 실제 사용자 데이터(CrUX)도 같이 보여준다
- 방문자가 없는 사이트는 시뮬레이션만 나와서 실질적 차이 없다. 그냥 DevTools Lighthouse 써도 된다.

## 4. Lab Data vs Field Data

---

PageSpeed Insights에서 결과를 보면 두 가지 섹션으로 나뉜다.

### 4.1 Lab Data

구글 서버에서 지금 시뮬레이션 돌린 결과다. 실제 사용자가 없어도 항상 나온다. LCP, TBT 같은 지표가 여기서 보인다.

### 4.2 Field Data

구글이 크롬 사용자들한테서 실제로 수집한 데이터다(CrUX). 진짜 사용자들이 28일간 경험한 평균값이라 훨씬 신뢰도가 높다.

CLS랑 INP는 Field Data에서만 제대로 나오는데, 이유가 있다.

- CLS는 페이지가 로딩되면서 요소가 실제로 움직여야 수치가 잡힌다
- INP는 사용자가 실제로 클릭을 해야 측정된다

시뮬레이션은 그냥 페이지 열고 스크롤만 하는 수준이라 이 두 개는 Lab Data에서 잘 안 나온다. 방문자가 없는 사이트에서 CLS랑 INP가 안 보이는 건 오류가 아니라 정상이다.
