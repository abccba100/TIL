## 1. Feature-based


### 1.1 기본 개념

Featrue-based는 기능(도메인) 단위를 코드를 묶는 방식으로 UI, api 함수, 특정 로직을 수행하는 함수 등을 한 폴더에 묶어 응집도를 높이는 아키텍처이다.

- 어쩌다가 Feature-based 아키텍처가 등장했을까 궁금해서 찾아보니
    
    Feature-based 아키텍처는 Layer-based 아키텍처를 보완하기 위해서 등장했다고 한다.  
    그래서 Layer-based 아키텍처에서 겪던 문제들 중 하나인 낮은 응집도를 높이기 위해 초점을 맞춘 것 같은 느낌이 많이 든다.
    

### 1.2 구조

아래는 Feature-based의 예시 구조이다.

`app/`, `router/`, `pages/`는 다른 아키텍처들과 비슷하거나 같은 역할을 한다.  
주요 특징은 위에서 말했던 것처럼 `featres/` 아래에 기능(도메인) 별로 폴더가 나누어지고,  
각 폴더 안에서는 해당 feature 전용 UI, API 함수, utill 함수, hook 등을 정의하여 사용한다.  

```
src/
├─ features/
│  ├─ auth/
│  │  ├─ components/     <- 해당 feature 전용 UI
│  │  ├─ hooks/          <- 해당 feature 전용 hook
│  │  ├─ api.ts          <- 해당 feature 전용 API 호출 로직
│  │  ├─ types.ts        <- 해당 feature 전용 타입
│  │  └─ utils/          <- 해당 feature 전용 utill 함수
│  └─ payment/
│      └─ (동일 구조)
├─ shared/
│  ├─ components/        <- Button, Input 등 공통 UI
│  ├─ constants/
│  ├─ hooks/
│  └─ utils/
├─ pages/                 <- 여러 feature를 조합한 페이지
├─ router/                <- 또는 routes/
└─ app/                   <- 전역 설정 (Provider, 전역 스타일 등)
```

### 1.3 규칙

사실 원래는 별도의 규칙이 없다.  
FSD 아키텍처는 규칙이 빡빡하지만, Feature-based 아키텍처는 그 전에 나온 아키텍처이기 때문에 별도의 규칙이 없다. 그래서 다른 사람이 세워둔 규칙들을 보고 와닿는 것을 적어보자면 다음과 같다.

1. **Feature는 서로를 참조하지 않는다.**
    
    위에서 말한 예시 구조로 설명하자면, `auth Feature`에서 `pyment Feature`의 내부 로직을 참조하지 않는 것을 말한다.  
    Feature들이 서로를 참조하게 되면 기능(도메인) 별로 나누어서 코드를 작성하는 것이 무의미해지고,  
    Feature 간의 경계가 사라질 수 있다. 그래서 공통으로 사용되는 로직은 `feature`에서 `shared` 로 옮겨야 한다.
    
2. **공통 코드는 shared에만 존재하며, 도메인 개념을 포함하지 않는다.**
    
    공통으로 사용되는 로직은 위에서 말한 이유 때문에 `shared` 에 존재해야 한다.  
    `shared` 는 여러 `feature` 에서 사용함으로 도메인 개념을 포함하지 않고 순수해야 한다.
    
3. **의존성은 `shared` → `feature` → `app`  방향으로만 흐른다.**
    
    `feature` 는 `shared` 를 사용할 수 있고,  
    `app` 은 `feature` 를 사용할 수 있다.  
    단, `feature`가 `app`을 참조하거나, `shared`가 `feature` 를 참조하는 등 반대 방향의 의존성은 허용하지 않는다.
    

위에서 말한 것 대부분 FSD 아키텍처의 규칙들과 비슷해 보이는데, 어쩌면 당연하다고 생각든다.  
feature-based 아키텍처의 부족한 부분을 해결하기 위해 FSD 아키텍처가 나왔으니 feature-based 아키텍처를 유용하게 사용하기 위해서는 FSD 아키텍처의 규칙을 가져올 수 밖에 없다고 생각된다.

## 2. Layer-based, FSD와의 관계 및 회고


### 2.1 Layer-based와의 관계, 회고

우선 Layer-based 아키텍처를 먼저 보면, Layer-based는 처음 프로젝트를 시작할 때 비교적 쉽고 사용하기 간편하다는 이유로 인해서 많이 사용했다.  
하지만, 프로젝트가 커지기 시작하니 처음에는 의도하지 않았던 특정 기능(도메인)의 로직이나 UI 등이 다른 곳에서도 계속 쓰이게 됐다. 어쩌면 그때의 나에게는 당연했을 것 같다.  
”코드를 재사용은 당연한 거니까” 라는 생각으로 특정 도메인의 내부 로직을 여러 곳에서 사용하도록 했다.  

그러다 문제가 터졌다 디자인 및 기능 수정에 맞춰 코드를 수정하니 내가 예상하지 못한 곳에서 문제들이 터졌다. UI가 꺠지고, 동작이 정상적으로 수행되지 않는 심각한 문제들이 터져버렸다.  
원인 당연하게도 “코드를 재사용은 당연한 거니까”라는 생각으로 특정 도메인의 내부 로직을 여러 곳에서 사용했던 것이다.  

결론적으로는 이 당시 시간도 없었고 서로 엮여있는 코드들이 많아서 유지보수를 포기하기 “일단 돌아가는 코드”로 작성해버렸다.  
이 과정에서 수 많은 중복 코드들이 생겨났고 이후 개발은 점점 힘들어졌다.  
이걸 뼈저리게 느낀 게 RN으로 진행했던 책마루 프로젝트다.  

결론적으로 Layer-based 아키텍처와의 관계는 ‘기능(도메인)에 따른 분리’라고 생각한다.  
때문에 Layer-baed에서 기능(도메인)을 잘 분리하면 feature-based와 큰 차이가 없다고 생각한다. (응집도 문제는 여전하겠지만)  
어쩌면 거의 같다는 생각이 든다.  

### 2.2 FSD와의 관계, 회고

위에서 말했던 것처럼 Layer-baed에서 여러 문제들(결국 내 문제가 컸지만)을 겪고 눈을 돌린 게 FSD 였다.  
그래서 FSD를 공부해보고 적용시켜 볼까 생각도 해봤지만, 잘 와닿지 않았다.  
이유는 내가 진행하는 프로젝트에서 FSD를 적용 시킬 만큼 큰 프로젝트는 없었고, 작은 프로젝트에 억지로 적용시키자니 큰 이점을 얻지 못하고 오히려 복잡성만 늘어날 것 같다고 판단했다.  

이후 기업 프로젝트의 엄청나게 큰 코드 베이스에 적용된 FSD를 보고 납득했다.  

어쨋든 이번에 프로젝트에서 적용할 아키텍처를 찾다보니 Layer-based와 FSD의 중간? 아키텍처인 Feature-based에 눈을 돌리게 됐다.  

결론적으로 FSD 아키텍처와의 관계는 ‘규칙과 빡빡함’이라고 생각한다.  
때문에 Feature-based에 위에서 말했던 것과 같은 규칙들을 추가하면 FSD까지는 아니어도 비슷한 수준의 이점들을 얻을 수 있다고 생각한다.  


---
이번에 Feature-based 아키텍처를 공부하다 보니 전에 잊고 있던 것들도 생각나고 어떤 아키텍처를 적용시킬 것인지에 대한 고민도 많이 하게 되었다.  
결론적으로 이번에 새롭게 시작할 프로젝트에선는 Feature-based 아키텍처에 규칙을 추가하여 진행해 볼까라는 생각이 들었다.  
이유는 프로젝트를 처음 구성하기에 부담이 없고, 추후 FSD로 바꾼다고 하더라도 사실상 폴더 및 파일만 분리하는 수준으로 끝날 것 같다고 결론을 내렸기 때문이다.
