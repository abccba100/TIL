## 1. 컴포넌트 테스트


### 1.1 개념

컴포넌트가 올바르게 렌더링되는지, 유저 인터랙션에 맞게 동작하는지 검증하는 테스트다.

Unit 테스트가 순수 함수를 검증한다면, 컴포넌트 테스트는 UI 단위를 검증한다.

## 2. RTL (React Testing Library)


### 2.1 개념

컴포넌트 테스트의 표준 도구다. 라이브러리 이름은 `@testing-library/react`이다.

RTL은 테스트 러너가 아니라 컴포넌트를 DOM에 올리고 쿼리하는 도구라서 Vitest나 Jest 위에서 같이 써야 한다.

### 2.2 핵심 철학

테스트는 소프트웨어가 사용되는 방식과 유사해야 한다.

내부 구현(state, props)이 아니라 유저 관점에서 테스트한다. 버튼이 있으면 클릭하고, 텍스트가 있으면 읽는 식으로.

## 3. 같이 쓰는 라이브러리


### 3.1 목록

세 개는 각각 독립적인 개별 패키지지만 RTL 생태계 안에서 세트처럼 같이 쓰는 게 관례다.

- `@testing-library/react` — 핵심 패키지. 컴포넌트 렌더링 + DOM 쿼리 담당
- `@testing-library/user-event` — 실제 유저 동작에 가까운 이벤트 시뮬레이션. `fireEvent`보다 정교함
- `@testing-library/jest-dom` — `toBeInTheDocument()`, `toHaveValue()` 같은 DOM 특화 matcher를 Jest/Vitest에 추가해주는 확장 패키지

## 4. 주요 API


### 4.1 개념

여기서 API는 HTTP 통신이 아니라, 라이브러리가 외부에 제공하는 함수/객체의 인터페이스를 뜻한다.

RTL을 설치하면 쓸 수 있는 함수들이 곧 RTL의 API다.

### 4.2 목록

- `render` — 컴포넌트를 DOM에 렌더링
- `screen` — 렌더링된 DOM에 접근하는 객체
- `getByText`, `getByRole` — 요소를 찾는 쿼리
- `fireEvent` / `userEvent` — 클릭, 입력 같은 이벤트 발생
- `expect` — Jest/Vitest assertion 그대로 씀

## 5. 쿼리 우선순위


### 5.1 권장 순서

RTL은 요소를 찾는 방법에 우선순위가 있다.

1. `getByRole` — 제일 권장 (접근성 기준)
2. `getByLabelText` — input 같은 폼 요소
3. `getByText` — 텍스트로 찾기
4. `getByTestId` — 최후의 수단

`getByRole`을 권장하는 이유는 접근성(a11y)과 테스트를 동시에 챙길 수 있어서다.

## 6. Enzyme과의 차이


### 6.1 Enzyme이란

Enzyme은 Airbnb가 만든 React 컴포넌트 테스트 라이브러리로, RTL 이전의 표준이었다. 내부 state나 props에 직접 접근해서 검증하는 방식이었다.

### 6.2 RTL과의 차이

Enzyme은 구현 세부사항을 테스트하고, RTL은 유저 관점을 테스트한다.

예를 들어 버튼 클릭 후 `count` state가 1 증가했는지 확인하는 건 Enzyme 방식이고, 버튼 클릭 후 화면에 숫자가 1 올라갔는지 확인하는 건 RTL 방식이다. 요즘은 RTL이 사실상 표준이다.
