## 1. 왜 테스트를 쓰는가


### 1.1 목적

코드를 고쳤는데 다른 데가 조용히 망가지는 걸 방지하려고 쓴다.

기능 추가하다 보면 예전 로직이 아무 에러 없이 그냥 망가지는 경우가 생긴다. 테스트가 있으면 그걸 바로 잡아준다.

## 2. 테스트 피라미드


### 2.1 개념

테스트를 3단계 레이어로 나눈 모델이다. 아래로 갈수록 많이 쓰고, 위로 갈수록 적게 쓰는 게 권장 비율이다.

- **Unit** — 함수 하나, 컴포넌트 하나 단위로 격리해서 검증한다. 외부 의존 없이 독립적으로 돌아가고, 제일 빠르다.
- **Integration** — 여러 모듈이 붙었을 때 잘 맞물리는지 확인한다. 버튼 누르면 API 호출되고 상태 바뀌는 흐름 같은 것.
- **E2E** — 실제 브라우저 띄워서 유저처럼 처음부터 끝까지 돌린다. 제일 현실에 가깝지만 느리고 깨지기 쉬워서 핵심 시나리오만 쓴다.

권장 비율은 대략 Unit 70 / Integration 20 / E2E 10이다.

## 3. 각 레이어 도구


### 3.1 도구 목록

- Unit → `Vitest` (Vite 쓰면 Vitest가 표준), `Jest`
- Integration → `React Testing Library (RTL)`
- E2E → `Playwright` (요즘 대세), `Cypress`

## 4. 언제 어떤 레이어를 쓸지


### 4.1 판단 기준

얼마나 많은 걸 묶어서 테스트하냐로 판단한다.

- 로직이 있는 순수 함수 → Unit
- 컴포넌트가 인터랙션에 맞게 동작하는지 → Integration
- 로그인하고 게시글 쓰고 저장되는 전체 흐름 → E2E

## 5. E2E를 적게 쓰는 이유


### 5.1 트레이드오프

E2E는 현실에 제일 가깝지만 그만큼 비용이 크다.

- **느림** — 브라우저 실제로 띄우고 서버도 떠야 해서 Unit 대비 수십 배 느리다.
- **flaky** — 네트워크 타이밍, 애니메이션, 로딩 순서 때문에 코드 안 바꿨는데 테스트가 가끔 실패한다.
- **유지보수 힘듦** — UI 조금만 바뀌어도 테스트가 깨진다. 버튼 텍스트 하나 바꿨는데 E2E 10개 고쳐야 하는 상황이 온다.

그래서 진짜 핵심 흐름만 E2E로 커버하고 나머지는 아래 레이어에서 잡는 게 맞다.
